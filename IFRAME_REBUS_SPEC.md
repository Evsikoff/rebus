# Спецификация: iframe-ребусы

Документ описывает протокол интеграции внешних веб-приложений (iframe-ребусов) с родительским приложением «Советские Ребусы».

---

## Концепция

Некоторые ребусы требуют сложного интерактивного интерфейса (перетаскивание, анимации, многошаговые головоломки), который невозможно реализовать средствами встроенного конструктора. Такие ребусы оформляются как отдельные веб-приложения и встраиваются через `<iframe>` в модальное окно игрового экрана.

**В конструкторе** для таких ребусов указывается только поле «iframe URL». Поля «Отгадка», «Изображения» и «Текстовое задание» не используются.

---

## Жизненный цикл

```
Пользователь кликает на плитку
        │
        ▼
Родитель открывает iframe-модалку, устанавливает src
        │
        ▼
iframe загружается → родитель отправляет rebus:init
        │
        ├─▶ Пользователь взаимодействует
        │         │
        │         ├─▶ [опционально] дочернее приложение отправляет rebus:started
        │         ├─▶ [по мере работы]  дочернее приложение отправляет rebus:progress
        │         └─▶ [при успехе]      дочернее приложение отправляет rebus:solved
        │
        └─▶ Пользователь закрывает модалку (кнопка ✕, Escape, клик вне окна)
                  │
                  └─▶ [опционально] дочернее приложение отправляет rebus:close
```

---

## Протокол обмена сообщениями (postMessage)

Все сообщения передаются через `window.postMessage`. Формат — объект JSON с обязательным полем `type`.

### Родитель → Дочернее приложение

#### `rebus:init`

Отправляется **один раз**, сразу после загрузки iframe. Передаёт контекст ребуса.

```json
{
  "type": "rebus:init",
  "rebusId": "2725b9c8",
  "solved": false,
  "progress": null
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `rebusId` | `string` | Уникальный идентификатор ребуса |
| `solved` | `boolean` | `true`, если пользователь уже решил этот ребус ранее |
| `progress` | `object \| null` | Ранее сохранённый промежуточный прогресс (произвольный JSON, ранее переданный через `rebus:progress`) |

**Рекомендуемое поведение дочернего приложения:**
- Если `solved === true` — показать экран «уже решено» или финальное состояние без возможности повторного зачёта.
- Если `progress !== null` — восстановить состояние игры из переданных данных.

---

### Дочернее приложение → Родитель

Все исходящие сообщения отправляются через:

```javascript
window.parent.postMessage({ type: "...", ... }, "*");
```

---

#### `rebus:started`

Пользователь начал взаимодействовать с ребусом (например, нажал «Начать», сделал первый ход).

```json
{
  "type": "rebus:started"
}
```

**Эффект в родителе:** плитка ребуса переходит в состояние «В процессе» (если ещё не решена).

> Это сообщение опционально — родитель переводит плитку в «В процессе» уже при открытии модалки. Но его можно использовать для более точного отслеживания момента начала.

---

#### `rebus:solved`

Пользователь успешно решил ребус.

```json
{
  "type": "rebus:solved"
}
```

**Эффект в родителе:**
- Плитка переходит в состояние «Решено» (красная с золотой звездой).
- Счётчик решённых ребусов обновляется.
- Факт решения записывается в `localStorage` с отметкой времени.

> Отправляйте это сообщение **только один раз** при первом успешном завершении. Повторные отправки не сломают систему, но будут зачтены как отдельные решения в статистике.

---

#### `rebus:progress`

Сохранение промежуточного состояния игры. Вызывайте при каждом значимом изменении.

```json
{
  "type": "rebus:progress",
  "data": {
    "step": 3,
    "selectedPieces": [1, 4, 2],
    "timeSpent": 142
  }
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `data` | `object` | Произвольный JSON — структуру определяет разработчик дочернего приложения |

**Эффект в родителе:** данные сохраняются в `localStorage` под ключом `rebus_iframe_progress_{rebusId}` и будут переданы обратно через `rebus:init.progress` при следующем открытии.

> **Ограничение:** объём данных ограничен квотой `localStorage` (~5 МБ). Не сохраняйте изображения или большие бинарные данные.

---

#### `rebus:close` (опционально)

Дочернее приложение просит закрыть модальное окно (например, после нажатия кнопки «Закрыть» внутри ребуса).

```json
{
  "type": "rebus:close"
}
```

**Эффект в родителе:** модалка закрывается, игровой экран перерисовывается.

---

## Пример реализации дочернего приложения

```javascript
// Получить init от родителя
window.addEventListener("message", (event) => {
  const { type, rebusId, solved, progress } = event.data || {};
  if (type !== "rebus:init") return;

  if (solved) {
    showSolvedScreen();
    return;
  }

  if (progress) {
    restoreState(progress);
  } else {
    startFresh();
  }
});

// Сообщить о начале
function onFirstInteraction() {
  window.parent.postMessage({ type: "rebus:started" }, "*");
}

// Сохранить прогресс (вызывать при каждом ходе)
function onStateChange(state) {
  window.parent.postMessage({
    type: "rebus:progress",
    data: state
  }, "*");
}

// Сообщить об успешном решении
function onPuzzleSolved() {
  window.parent.postMessage({ type: "rebus:solved" }, "*");
  showVictoryAnimation();
}
```

## Рекомендации по UX

| Рекомендация | Обоснование |
|---|---|
| Показывать собственную анимацию победы перед `rebus:solved` | Пользователь видит подтверждение внутри ребуса, а не только смену плитки снаружи |
| Вызывать `rebus:progress` не чаще раза в секунду | Избежать перегрузки localStorage в длинных сессиях |
| При `solved === true` из `rebus:init` блокировать повторное прохождение | Иначе пользователь может «решить» ребус несколько раз, что исказит статистику |
| Адаптивная верстка (100% ширины и высоты) | iframe занимает всё модальное окно, которое меняет размер на мобильных |
| Не показывать собственную кнопку «Закрыть» поверх заголовка модалки | Кнопка ✕ уже есть в родителе; если нужна своя — размещайте внизу или сбоку |

---

## Безопасность

- Родитель фильтрует входящие `postMessage` по `event.origin`: принимаются только сообщения с домена, совпадающего с `origin` URL из поля `iframeUrl`.
- Дочернее приложение может передавать сообщения на `"*"` — родитель сам проверяет origin.
- Не передавайте через `postMessage` конфиденциальные данные (токены, пароли).
- iframe загружается с атрибутом `allow="fullscreen"` и без `sandbox` для максимальной совместимости. При необходимости ужесточения политики — добавьте `sandbox` в атрибут iframe в `menu.html`.

## Схема данных ребуса в rebuses.json

```json
{
  "id": "abc12345",
  "order": 0,
  "iframeUrl": "https://my-puzzle-app.vercel.app/rebus/1",
  "answer": "",
  "images": [],
  "texts": [],
  "explanation": ""
}
```

Если поле `iframeUrl` не пустое — игровой экран игнорирует `answer`, `images`, `texts` и открывает iframe-модалку вместо стандартной.
